**Role:** You are a Senior SDET specializing in C99 robustness testing — resource leaks, null safety, and invalid state handling.

**Context:**
The JZ-HDL compiler is a C99 application using region-based allocation (`JZArena`), a global compiler context (`JZCompiler`), and multiple processing phases (lexer → parser → semantic analysis → IR build → backend). You are writing C test programs that link against the compiler's internal APIs to verify the compiler does not leak memory, crash on null/invalid inputs, or leave itself in an inconsistent state after errors.

**Prerequisites:**
- Read `pipeline/dependancies.md` for the dependency graph, leaf/root nodes, and side-effect inventory.
- Read the public headers in `compiler/include/` to understand each module's API surface.
- Read `compiler/src/arena.c`, `compiler/src/compiler.c`, and `compiler/src/diagnostic.c` to understand resource lifecycle.

**Task:**
Generate a test plan and corresponding C test files for two categories of compiler robustness bugs.

---

### Category 1: Resource Leaks

**Goal:** Verify that every code path through the compiler properly frees all allocated resources.

**What to test:**

1. **Arena lifecycle** — For each arena (`ast_arena`, `ir_arena`):
   - Normal path: init → use → dispose releases all blocks.
   - Error path: init → parse failure → dispose still releases all blocks.
   - Reset path: init → use → reset → use → dispose is clean.
   - Zero-size allocation: `jz_arena_alloc(arena, 0)` does not leak or corrupt.

2. **Compiler context lifecycle** — Using `JZCompiler`:
   - Normal path: init → parse → semantic → IR → dispose is clean.
   - Early abort: init → parse fails → dispose is clean (no IR built).
   - Double dispose: init → dispose → dispose does not crash or double-free.
   - Multiple compilations: init → compile → dispose → init → compile → dispose (no cumulative leaks).

3. **Diagnostic list** — Using `JZDiagnosticList`:
   - Adding diagnostics then disposing the compiler frees all diagnostic strings.
   - Adding zero diagnostics then disposing is clean.
   - Adding diagnostics during parse failure then disposing is clean.

4. **File handle leaks** — For any module that opens files (lexer reading source, `@import`, mem init files, chip data JSON):
   - Normal path: file opened and closed.
   - File not found: no handle leaked.
   - Parse error mid-file: file still closed.

5. **Per-phase leak testing** — Using the dependency graph from `pipeline/dependancies.md`, identify the leaf nodes (no internal dependencies) and test each in isolation:
   - Allocate inputs, call the function, verify all outputs are accounted for, dispose.
   - Use the "Side Effects & Mocking Requirements" table to identify which modules perform allocation as a side effect.

**How to test:**
- Write C test programs that call internal APIs directly.
- Use a wrapper around `malloc`/`free` that counts allocations and frees — verify the count is zero after each test.
- Alternatively, run tests under Valgrind/AddressSanitizer and check for leak reports.
- Each test should be a function that returns 0 on success or non-zero on failure.

---

### Category 2: Null/Invalid State Handling

**Goal:** Verify that every public API function handles null pointers and invalid state without crashing (no segfault, no undefined behavior).

**What to test:**

1. **Null pointer arguments** — For every public function in `compiler/include/*.h`:
   - Call with NULL for each pointer parameter (one at a time, others valid).
   - Expected behavior: returns an error code, returns NULL, or is a documented no-op. Must NOT crash.

2. **Uninitialized compiler context:**
   - Call `jz_compiler_dispose()` on a zero-initialized `JZCompiler` (not `jz_compiler_init`'d).
   - Call parse/semantic/IR functions on an initialized but unused compiler context.

3. **Invalid phase ordering:**
   - Call semantic analysis before parsing (no AST exists — `ast_root` is NULL).
   - Call IR build before semantic analysis.
   - Call backend before IR build (`ir_root` is NULL).
   - Each should produce a diagnostic or return an error, not crash.

4. **Corrupt/truncated input:**
   - Pass an empty string to the lexer.
   - Pass a string that ends mid-token (e.g., `"@modul"` truncated).
   - Pass a file path that is a directory, not a file.
   - Pass binary garbage as source input.
   - Pass an extremely long line (>64KB) or extremely deep nesting (>1000 levels).

5. **AST/IR node null fields** — Using the dependency graph, identify intermediate nodes that consume AST/IR structures:
   - Construct an AST node with NULL children where children are expected.
   - Pass it to semantic analysis — verify it doesn't dereference NULL.
   - Construct an IR node with NULL fields and pass to the backend.

6. **Diagnostic overflow:**
   - Add an extremely large number of diagnostics (>100,000) — verify no OOM crash, or graceful handling.

2. Read all public headers in `compiler/include/` to enumerate every function that takes a pointer parameter.

6. Run the tests and report results. For leak detection, note which tests should be run under AddressSanitizer (`-fsanitize=address`) or Valgrind.
