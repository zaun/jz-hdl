If a validation test already exists for a rule ID, **do not overwrite it** — skip that rule and note it in your output summary.

### `.jz` File Structure:

Every `.jz` test file MUST follow this structure:

```jz
// <RULE_ID>: <brief description of what is being tested>.
// Each trigger is commented. Valid uses are included to verify no false positives.

@project <RULE_ID>_<TEST_NAME>
    CONFIG {
        WIDTH = 8;
    }

    @top <TopModuleName> {
        IN  [1] clk = _;
        OUT [1] data = _;
    }
@endproj

// Trigger 1: <rule violation in context A, e.g., module name>
@module <BadName>
    PORT {
        IN  [1] clk;
        OUT [1] data;
    }

    REGISTER {
        r [1] = 1'b1;
    }

    ASYNCHRONOUS {
        data = r;
    }

    SYNCHRONOUS(CLK=clk) {
        r <= ~r;
    }
@endmod

// Top module — structurally valid, also triggers rule in contexts B and C
@module <TopModuleName>
    // Trigger 2: <rule violation in context B, e.g., CONST name>
    CONST {
        <BadName> = 1;
    }

    PORT {
        IN  [1] clk;
        OUT [1] data;
    }

    // Trigger 3: <rule violation in context C, e.g., instance port binding>
    @new inst_a <BadName> {
        IN  [1] clk = _;
        OUT [1] data = _;
    };

    @new inst_b <HelperModule> {
        // Trigger 4: <rule violation in context D, e.g., port name in binding>
        IN  [1] <BadName> = _;
        OUT [1] data = _;
    };

    REGISTER {
        r [1] = 1'b1;
    }

    ASYNCHRONOUS {
        data = r;
    }

    SYNCHRONOUS(CLK=clk) {
        r <= ~r;
    }
@endmod

// Trigger 5: <rule violation in context E, e.g., port declaration name>
@module <HelperModule>
    PORT {
        IN [1] <BadName>;
        OUT [1] data;
    }

    REGISTER {
        r [1] = 1'b1;
    }

    ASYNCHRONOUS {
        data = r;
    }

    SYNCHRONOUS(CLK=<BadName>) {
        r <= ~r;
    }
@endmod
```

**This template demonstrates the key principles:**
- Multiple modules that reference each other via `@new`
- The rule is triggered in every distinct syntactic context it can appear in
- Valid constructs surround the triggers (e.g., valid port names, valid `_` no-connect uses)
- The `@top` module is structurally sound
- Each trigger has an explanatory comment

### Context Enumeration (critical step before writing any test)

Every compiler rule can fire in one or more syntactic contexts. Before writing a test, you MUST enumerate every context where the rule under test can trigger. Think about:

- **Declaration contexts:** Where can the violating construct appear as a declaration? (module name, const name, port name, register name, wire name, etc.)
- **Reference contexts:** Where can it appear as a reference? (instance target, port binding, CLK parameter, expression operand, etc.)
- **Cross-module contexts:** Can the violation appear in an instance relationship? (instance port bindings, instance target module names, etc.)
- **Block contexts:** Which block types can contain the violation? (ASYNCHRONOUS, SYNCHRONOUS, CONST, PORT, REGISTER, WIRE, etc.)

**Your test MUST trigger the rule in as many of these applicable contexts as possible.** A test with only 1-2 triggers when the rule can fire in 4-5+ contexts is incomplete and unacceptable — it misses the majority of the coverage. The goal is to verify the compiler catches the violation everywhere it can occur, not just in one convenient spot.

---

**Structural requirements:**

1. **`@project` block is mandatory.** Every test file must have a valid `@project` wrapper with `CONFIG`, and a `@top` declaration that binds to one of the modules in the file.
2. **The `@top` module must be structurally valid.** It should compile cleanly (no errors of its own) so that only the intentional rule violations produce diagnostics.
3. **Multiple triggers per file across diverse contexts — this is the most important requirement.** Each `.jz` file MUST test the target rule in as many distinct syntactic positions as the rule can appear in. Refer to the Context Enumeration section above — enumerate every applicable context, then include a trigger for each one. If a rule can fire in 5 contexts, the test must have at least 4-5 triggers. A test file with only 1-2 triggers when the rule can fire in more contexts is a **test quality bug** — go back and add more triggers. Add a comment above each trigger explaining what context it tests.
4. **Multi-module interaction is mandatory for most tests.** Tests MUST use 2-3 modules that reference each other via `@new` instantiation. This exercises the rule through instance resolution and cross-module paths. The typical pattern is: one module triggers the violation in one context, the `@top` module instantiates it and triggers in another context, and a helper module triggers in yet another. Single-module tests are only acceptable when the rule physically cannot apply across modules.
5. **Implicit negative testing.** Where a construct has both valid and invalid uses, include the valid uses alongside the invalid ones in the same file. For example, if testing a rule that forbids something in one context but allows it in another, include both in the same test to verify the compiler does not false-positive on the valid uses.
6. **No cascading or unrelated errors.** The `.out` file must contain ONLY diagnostics for the rule under test. If your test file triggers unrelated rules (e.g., `TOP_PORT_NOT_LISTED`, `WARN_UNUSED_MODULE`, width mismatches), that is a bug in your test — restructure the `.jz` file to eliminate the unrelated diagnostics. Every diagnostic in the `.out` file should have the same rule ID as the test name (with rare, justified exceptions).
7. **Modules must be complete.** Every `@module` needs at minimum: a `PORT` block with at least one `IN` and one `OUT`, and an `ASYNCHRONOUS` block that connects them (or a `REGISTER` + `SYNCHRONOUS` block if clocked behavior is needed). Modules that reference registers must include `REGISTER` and `SYNCHRONOUS` blocks.
8. **Module instantiation with `@new` requires the target module to exist in the file.** If testing `@new`, define the instantiated module in the same file.
9. **Use `_` (no-connect) for port bindings in `@top` and `@new` where the connection is not relevant to the test.**
10. **Do not introduce unrelated errors.** The test must compile cleanly except for the specific rule being tested. Avoid undeclared identifiers, width mismatches, or other incidental errors. After writing each test, mentally compile it and verify no unrelated diagnostics would fire.

### `.out` File Format:

The `.out` file contains the exact expected output of `jz-hdl --info --lint <file>.jz`. The format is:

```
File: <FILENAME>.jz
      <line>:<col>    <severity> <RULE_ID> <message>
```

**Format rules:**

1. **First line** is always `File: <FILENAME>.jz` (filename only, no path).
2. **Each diagnostic** is on its own line, indented with 6 spaces, formatted as:
   - `<line>:<col>` — left-aligned within a field, followed by spaces to align
   - `<severity>` — `error`, `warning`, or `info` (matches `rules.c` mode)
   - `<RULE_ID>` — exact rule ID from `rules.c`
   - `<message>` — exact message string from `rules.c`
3. **Diagnostics are ordered by line number**, then column number.
4. **The file ends with a trailing newline.**
5. **Line and column numbers must exactly match the `.jz` source.** Count carefully:
   - Line numbers are 1-based.
   - Column numbers are 1-based, counting from the start of the line to the first character of the offending token.
   - For `@module <Name>`, the module name token starts at the column after `@module `.
   - For `CONST { <Name> = ...`, the const name starts at its indented column position.
   - For port declarations like `IN [1] <Name>;`, the port name starts at its column position.

---

### Determining Line and Column Numbers:

Since you are writing the `.jz` file, you control the layout. Follow this process:

1. Write the complete `.jz` file first.
2. Number every line starting from 1.
3. For each intentional rule violation, identify the exact line and count the column of the offending token (1-based, counting each character including spaces).
4. Record these in the `.out` file.

**Common column positions** (given 4-space indentation):
- `@module NAME` → NAME starts at column 9
- `@module` at file scope → `@module` starts at column 1, so the module name starts at column 9
- `CONST { NAME = ...` with 8-space indent → NAME starts at column 9
- `IN  [1] NAME;` with 8-space indent → depends on exact spacing; count carefully
- Port name in `@new` binding like `IN  [1] NAME = _` with 8-space indent → count from line start

### Quality Checklist (verify EVERY test file against this list before writing the `.out` file):

For each `.jz` test file, answer these questions. If any answer is "no", fix the test before proceeding:

1. **Trigger count:** How many triggers does this test have? Count them. If the rule can fire in N syntactic contexts and your test has fewer than N-1 triggers, the test is incomplete. Go back and add triggers for the missing contexts.
2. **Context diversity:** List the contexts your triggers cover (e.g., "module name, const name, port decl, instance binding"). Compare against your context enumeration from the workflow. Are you missing any applicable context? If yes, add it.
3. **Multi-module structure:** Does the file have 2+ modules connected via `@new`? If not, why not — does the rule truly only apply within a single module?
4. **No unrelated diagnostics:** Mentally compile the file. Will any rule OTHER than the one under test fire? Check for: `TOP_PORT_NOT_LISTED` (port exists in module but not in `@top`), `WARN_UNUSED_MODULE` (module defined but never instantiated or used as `@top`), width mismatches, undeclared identifiers. If any would fire, restructure to eliminate them.
5. **Negative testing:** Do valid uses of the construct appear alongside invalid uses? The valid uses must NOT appear in the `.out` file.
6. **Complete modules:** Does every module have PORT (IN + OUT), and either ASYNCHRONOUS or REGISTER+SYNCHRONOUS blocks? Are all ports connected?

### Happy-Path Tests:

Happy-path tests verify that valid code compiles without false positives. They are important regression tests.
- Create a `.jz` file with valid code that exercises the construct being tested.
- Create a `.out` file containing ONLY the `File:` header line and a trailing newline (no diagnostics).
- Name them with the convention: `<section>_<subsection>_<RULE_ID>-<TEST_NAME>_ok.jz`

- **Rules listed in the test plan's "Rules Missing" section** — these have no rule ID in `rules.c` and cannot be tested.
