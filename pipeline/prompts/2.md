**Role:** You are a Senior SDET specializing in compiler validation for a hardware description language.

**Context:**
I am building a compiler for JZ-HDL, a hardware correctness language. The compiler enforces strict semantic rules at compile time. I need a comprehensive test plan that ensures every rule the compiler checks is validated.

There are two key documents:
1. **The Language Specification** (`specification/jz-hdl-specification.md`): The source of truth for all language requirements, semantics, and constraints.
2. **The Rule ID Registry** (`compiler/src/rules.c`): Contains every concrete diagnostic rule ID the compiler implements (e.g., `CLOCK_GEN_MISSING_INPUT`, `BUS_SIGNAL_UNDEFINED`). This is the completeness checklist.

**Task:**
Create/update a comprehensive **Test Plan** for every individual subsection identified in the Language Specification.

### Prerequisite:
- Review existing test plans in `pipeline/test_*.md`. Update them in place — do not delete and recreate. If a plan already covers a spec subsection, add missing rules and scenarios to it rather than starting from scratch. Only create new plan files for spec subsections that have no existing plan.
- Review existing tests in `compiler/tests/validation/` to understand the current baseline and avoid duplicating work.

### Test Framework:
Each validation test consists of a paired `.jz` and `.out` file:
- The `.jz` file contains JZ-HDL source code designed to trigger specific diagnostics.
- The `.out` file contains the expected diagnostic output from running `compiler/build/jz-hdl --info --lint <file>.jz`.
- **Happy-path tests** use an empty `.out` file (no diagnostics expected) to verify that valid code compiles cleanly without false positives.
- Tests should include both error cases AND valid negative cases (code that should NOT trigger the rule) in the same file where practical.
- Tests targeting chip-specific rules (e.g., resource limits, pin standards) must include full project infrastructure (`CHIP=`, `IN_PINS`, `OUT_PINS`, `MAP`, `CLOCKS` blocks).

### Test Plan Structure:
For each spec subsection, provide:

1.  **Objective:** What specific language requirement is being verified?
2.  **Test Scenarios:**
    *   **Happy Path:** Valid code that must compile without diagnostics (empty `.out`). These are regression tests ensuring the compiler does not reject valid constructs.
    *   **Error Cases:** Code that violates the spec and must produce specific diagnostics.
    *   **Edge Cases:** Boundary conditions, unusual but valid constructs, scope interactions across module hierarchies.
3.  **Input/Output Matrix:**
    *   For each test scenario: the triggering construct, the expected rule ID, and the expected diagnostic severity (error/warning/info).
4.  **Rules Matrix:**
    *   **Rules Tested:** Every rule ID from `rules.c` that maps to this spec subsection, with at least one test scenario per rule.
    *   **Rules Not Tested:** Any rule ID that cannot be tested in the `--info --lint` framework, with a documented reason (e.g., fires during backend only, dead code, requires simulation).

### Reconciliation Step (CRITICAL):
After generating all test plans, perform a completeness check:
1. Extract every rule ID from `compiler/src/rules.c`.
2. For each rule ID, verify it appears in at least one test plan's Rules Matrix (either "Rules Tested" or "Rules Not Tested" with a reason).
3. Any rule ID not accounted for must be:
   - **(a)** Added to the appropriate spec section's test plan, OR
   - **(b)** Documented in `pipeline/not_tested.md` with a reason (dead code, backend-only, untriggerable), OR
   - **(c)** Flagged as a potential spec gap (rule exists in compiler but has no corresponding spec requirement).
4. The final count of (tested + documented-untestable + flagged-gap) must equal the total rule count in `rules.c`. Report this count.

### Formatting:
Format the output in Markdown. Use headers for each subsection and tables for the Input/Output and Rules matrices.

### Documents to process:
- `specification/jz-hdl-specification.md` (primary — drives test plan structure and scenarios)
- `compiler/src/rules.c` (completeness checklist — every rule ID must be accounted for)
- `compiler/tests/validation/` (existing test baseline — do not duplicate)

### Output Location:
File name should be lowercase and use underscores for spaces.
```
pipeline/test_<section>_<subsection>-<plan_name>.md
pipeline/test_<section>_<subsection>_<subsubsection>-<plan_name>.md
```
