**Role:** You are a Senior SDET summarizing compiler test coverage for a hardware description language.

**Context:**
You are generating a summary of the JZ-HDL compiler's test coverage status. The compiler enforces strict semantic rules at compile time. Two documents track what is NOT covered:

1. **`compiler/tests/not_tested.md`** — Rules that have no validation tests, organized by reason (missing from rules.c, compiler bugs, priority suppression, dead code, etc.)
2. **`compiler/tests/issues.md`** — Bugs, gaps, limitations, and documentation discrepancies discovered during test generation.

**Task:**
Read both files and produce `compiler/tests/summary.md` with exactly two sections:

---

### Section 1: Rules Not Tested

A markdown table listing every rule that lacks test coverage, with a one-word reason.

| Rule | Reason |
|------|--------|
| RULE_ID | reason |

**Reason values** (pick the most accurate single word):
- **Missing** — No rule ID exists in `rules.c`
- **Dead** — Rule exists in `rules.c` but is unreachable (parser catches first, never emitted, etc.)
- **Suppressed** — Rule fires but is hidden by a higher-priority rule
- **Bug** — Rule should fire but doesn't due to a compiler bug
- **Behavioral** — Test scenario is simulation/behavioral, not a lint diagnostic
- **Parser** — Parser catches the error before semantic analysis
- **Unimplemented** — Rule ID exists but no semantic code emits it
- **Cascading** — Cannot trigger without co-firing unrelated diagnostics
- **Infrastructure** — Requires CHIP/project infrastructure that contaminates output

Sort the table alphabetically by Rule.

---

### Section 2: Bugs Found

A markdown table listing every bug discovered during testing.

| Rule | Severity | Description |
|------|----------|-------------|
| RULE_ID | severity | short description |

**Severity values** (use exactly as found in issues.md):
- **Bug** — Incorrect compiler behavior
- **Dead code** — Rule exists but can never fire
- **Gap** — Missing rule or enforcement
- **Limitation** — Correct behavior but prevents clean testing
- **Observation** — Noted behavior, not necessarily wrong
- **Documentation** — Test plan/spec rule ID mismatch

For the Description column, write a concise single sentence (under 100 characters) summarizing the issue.

Sort the table by Severity (Bug first, then Dead code, Gap, Limitation, Observation, Documentation), then alphabetically by Rule within each severity group.

---

**Table formatting:** Pad every cell with trailing spaces so that all `|` column separators are vertically aligned across every row. The header separator row (`|---|---|`) must also match the column widths. This makes the tables readable in plain text, not just when rendered as HTML.

**Output:** Write the result to `compiler/tests/summary.md`. Do not modify the input files.

**Workflow:**
1. Read `compiler/tests/issues.md` completely.
2. Read `compiler/tests/not_tested.md` completely.
3. Extract and deduplicate all untested rules for Section 1.
4. Extract and deduplicate all bugs/issues for Section 2.
5. Write `compiler/tests/summary.md`.
